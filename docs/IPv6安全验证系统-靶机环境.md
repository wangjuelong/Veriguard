> 来源：[`IPv6安全验证系统.md`](./IPv6安全验证系统.md)（招标文件条款汇编）
> 整理日期：2026 年 5 月
> 范围：按业务模块推导的 4 类靶机环境（边界防护 / 流量安全 / 主机安全 / 自定义验证），不含沙箱

## 0. 概览矩阵

| 靶机环境 | 攻击源（执行端） | 攻击目标（被攻击端） | 被验证设备 | 验证维度 | Veriguard 实现 |
| --- | --- | --- | --- | --- | --- |
| 边界防护 | 协作主机 Agent（`http_attack`） | 客户 Web 应用 / API | WAF / IPS | 拦截 + 检测 | PR-C ✅ |
| 流量安全 | 协作主机 Agent（`pcap_replay`）或流量回放设备 | 网络中任意 IP / 端口 | NTA / IDS | 检测 +（inline 时）拦截 | PR-D ✅ |

| 自定义验证 | 视用例类型而定（5 类）| 视用例 | 视用例 | 视用例 | PR-B/C/D ✅ |

## 1. 边界防护验证靶机

### 1.1 模块定位

验证客户网络边界部署的 **WAF / IPS** 等防护设备，对各类边界侧攻击（命令执行 / SQL 注入 / SSRF / XSS / SSTI / 上传绕过等 12 类）的真实防护效用。

### 1.2 靶机部署形态

| 角色 | 部署位置 | 形态 |
| --- | --- | --- |
| 攻击源 | 客户内网或 DMZ，能访问被验证 Web 应用 | 协作主机 Agent（Linux 主机，声明 `http_attack` capability）|
| 被攻击端 | DMZ / 业务区 | 客户 Web 应用 / 业务系统 / API 网关 |
| 被验证设备 | 业务区前的边界 | WAF / IPS |
| 告警侧 | 内网 SOC 区 | SIEM（汇聚 WAF / IPS 告警）|

### 1.3 网络拓扑

```
图例:  ╔══╗ 被验证设备   ┌──┐ 平台/Agent/靶机   [ SIEM ] 告警汇聚
       〔 〕 网络区域     ──►(N) 数据流第 N 步（对应数据流时序表）

  〔内网管理区〕                                  〔内网 SOC 区〕

  ┌──────────────────────────┐
  │  Veriguard 平台           │ ◄── (7) Agent 上报响应
  │  WebAttackExecutor        │ ◄── (8) 查询 SIEM 关联告警
  └──────────┬───────────────┘
             │ (1) 派发 inject
             │     capability=http_attack
             ▼
  ┌──────────────────────────┐
  │  协作主机 Agent            │
  │  Linux + curl             │
  │  http_attack capability   │
  └──────────┬───────────────┘
             │ (2) HTTP / HTTPS 请求
  ─ ─ ─ ─ ─ ─│─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
  〔DMZ / 业务区〕
             ▼
  ╔══════════════════════════╗ ──── (4) 产生告警 ────┐
  ║  WAF / IPS                ║                       ▼
  ║  (被验证设备)              ║              ┌──────────────────┐
  ╚══════════╤═══════════════╝              │     [ SIEM ]      │
             │ (3) 拦截 / 放行                │  (5) 告警汇聚     │
             ▼                              └──────────────────┘
  ┌──────────────────────────┐
  │  Web 应用靶机              │   (6) HTTP 响应回到 Agent
  │  CMS / OA / API           │       (沿步骤 2 路径反向)
  │  (被攻击端)                │
  └──────────────────────────┘

  (9) Veriguard 后端 → 计算验证结果（拦截维度 + 检测维度）
```

### 1.4 数据流时序

| 步骤 | 主体 | 动作 |
| --- | --- | --- |
| 1 | 用户 | 在编辑器配置 web_attack inject（method / url / headers / body / 期望状态码）|
| 2 | Veriguard 后端 | `WebAttackExecutor` 按 capability 选协作主机 Agent |
| 3 | 协作主机 Agent | 模板替换后发起 HTTP / HTTPS 请求 |
| 4 | WAF / IPS | 根据策略拦截或放行；产生告警 |
| 5 | SIEM | 接收 WAF / IPS 告警 |
| 6 | Web 应用 | （若未被拦截）返回响应 |
| 7 | 协作主机 Agent | 上报响应（状态码 / 头 / 体 / 耗时）到平台 |
| 8 | Veriguard 后端 | 查 SIEM 关联告警 |
| 9 | Veriguard 后端 | 计算验证结果（拦截维度 + 检测维度）|

### 1.5 验证维度

| 维度 | 评估方式 |
| --- | --- |
| **拦截** | 响应状态码 / 错误页特征 / 连接重置 / 请求超时 |
| **检测** | SIEM 关联告警匹配（WAF / IPS）|

## 2. 流量安全验证靶机

### 2.1 模块定位

验证客户网络边界部署的 **NTA / IDS** 等流量检测设备，对各类流量侧攻击（暴力破解 / 反弹 shell / 隐秘隧道 / 远控木马 / 信息泄露等 11 类）的真实防护效用。

### 2.2 靶机部署形态

| 角色 | 部署位置 | 形态 |
| --- | --- | --- |
| 攻击源 | 流量可注入的网卡位置（与被监听网段同 VLAN）| 协作主机 Agent（Linux 主机，声明 `pcap_replay` capability，root 或 `CAP_NET_RAW` + `CAP_NET_ADMIN`）|
| 备选攻击源 | 客户网络边界 | 客户既有流量回放设备（外部硬件，通过接口适配层接入）|
| 流量目的 | 任意（流量经过检测设备即可）| 业务网段中的任意 IP / 端口 |
| 被验证设备 | 网段边界（旁路或串联）| NTA / IDS |
| 告警侧 | 内网 SOC 区 | SIEM |
| pcap 文件 | Veriguard 平台 | MinIO 对象存储 |

### 2.3 网络拓扑

```
图例:  ╔══╗ 被验证设备   ┌──┐ 平台/Agent/靶机   [ SIEM ] 告警汇聚
       〔 〕 网络区域     ──►(N) 数据流第 N 步（对应数据流时序表）

  〔内网管理区〕

  ┌──────────────────────────┐
  │  Veriguard 平台           │ ── (1) 上传 pcap ──► ┌──────────────────┐
  │  PcapReplayExecutor       │                      │  MinIO            │
  │  ◄ (9) 上报回放结果        │                      │  pcap 对象存储     │
  │  ◄ (10) 查询 SIEM 告警     │                      └─────────┬────────┘
  └──────────┬───────────────┘                               │
             │ (3) 派发 (capability=pcap_replay)              │ (4) 下载 pcap
             ▼                                              │
  ┌──────────────────────────────────────────────┐ ◄────────┘
  │  协作主机 Agent (Linux + tcpreplay)             │
  │  pcap_replay capability                       │
  │  〔业务网段同 VLAN / 镜像口接入点〕                │
  └──────────┬───────────────────────────────────┘
             │ (5) tcpreplay -i eth0 [params]
  ─ ─ ─ ─ ─ ─│─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
  〔业务网段（IPv6 单栈）〕                            〔内网 SOC 区〕
             ▼
  ╔══════════════════════════╗ ─── (7) 产生告警 ───► ┌──────────────────┐
  ║  NTA / IDS                ║                       │     [ SIEM ]      │
  ║  (被验证设备)              ║                       │  (8) 告警汇聚     │
  ╚══════════════════════════╝                       └──────────────────┘

  注：(2) 用户在编辑器配置 pcap_replay inject
      (6) 流量经过检测设备   (9) Agent 清理临时 pcap 文件
      (11) Veriguard 后端 → 计算验证结果
```

### 2.4 数据流时序

| 步骤 | 主体 | 动作 |
| --- | --- | --- |
| 1 | 用户 | 上传 pcap 文件到 MinIO（独立子项目，本期 PR 仅 pcap_file_id 引用）|
| 2 | 用户 | 在编辑器配置 pcap_replay inject（pcap_file_id / interface / mode / rate）|
| 3 | Veriguard 后端 | `PcapReplayExecutor` 按 capability 选协作主机 Agent |
| 4 | 协作主机 Agent | 从 MinIO 下载 pcap 文件到本机临时目录 |
| 5 | 协作主机 Agent | 调 tcpreplay 在指定网卡注入流量 |
| 6 | 业务网段 | 流量经过 NTA / IDS |
| 7 | NTA / IDS | 产生告警 |
| 8 | SIEM | 接收 NTA / IDS 告警 |
| 9 | 协作主机 Agent | 清理临时 pcap 文件 + 上报回放结果（包数 / 时长 / 错误）|
| 10 | Veriguard 后端 | 查 SIEM 关联告警 |
| 11 | Veriguard 后端 | 计算验证结果 |

### 2.5 验证维度

| 维度 | 评估方式 |
| --- | --- |
| **检测** | SIEM 关联告警匹配（NTA / IDS）|
| **拦截** | 取决于检测设备部署方式：旁路监听只能观察、不能拦截；inline 模式可观察阻断 |

### 2.6 招标特殊约束

招标流量章节强制：**同一用例需支持包含多个端口不同的四元组**。当前 `NetworkTraffic` schema 是单组四元组，实施期需扩展为多元组 List（已在用例清单缺口表标注）。

## 3. 主机安全验证靶机

### 3.1 模块定位

验证客户主机部署的 **HIDS / EDR** 等主机安全产品，对各类主机侧攻击（反弹 shell / 提权 / 远控木马 / 痕迹清理 / 持久化等 12 类）的真实防护效用。

### 3.2 靶机部署形态

主机侧有 **2 种 Agent 路径**：

#### 路径 A：OpenAEV Agent 直驱（推荐）

| 角色 | 部署位置 | 形态 |
| --- | --- | --- |
| 攻击源 = 被攻击端 | 被验证主机自身 | OpenAEV Agent（Rust 常驻进程，声明 `command_exec` / `file_drop` 等 capability）|
| 被验证设备 | 同一主机或独立 EDR 控制台 | HIDS / EDR |
| Agent 部署角色 | — | service.admin / service.standard / session.admin / session.standard 4 种组合 |

#### 路径 B：借第三方 EDR Agent

| 角色 | 部署位置 | 形态 |
| --- | --- | --- |
| 攻击源 = 被攻击端 | 被验证主机自身 | 客户既有 EDR Agent（Caldera / Tanium / CrowdStrike / SentinelOne / Cortex XDR）|
| 平台 → Agent 通道 | 厂商 SaaS / 私部 API | EDR remote-script / RTR API |

### 3.5 数据流时序（路径 A）

| 步骤 | 主体 | 动作 |
| --- | --- | --- |
| 1 | 用户 | 在编辑器配置主机攻击 inject（Command / Executable / FileDrop payload）|
| 2 | Veriguard 后端 | 按主机 asset + Agent capability 选定目标 + Agent |
| 3 | OpenAEV Agent | 在主机执行命令（PowerShell / bash / sh / cmd 或落盘 + 执行二进制）|
| 4 | 主机 | 产生事件（文件写入 / 进程创建 / 网络连接等）|
| 5 | HIDS / EDR | 监控事件并产生告警 |
| 6 | SIEM | 接收告警 |
| 7 | OpenAEV Agent | 上报命令输出 / 退出码到平台 |
| 8 | Veriguard 后端 | 查 SIEM 关联告警 |
| 9 | Veriguard 后端 | 计算验证结果 |

### 3.6 数据流时序（路径 B）

| 步骤 | 主体 | 动作 |
| --- | --- | --- |
| 1 | 用户 | 编辑 inject 并指定 executor（如 tanium / crowdstrike）|
| 2 | Veriguard 后端 | 调厂商 API 投递 remote-script |
| 3 | 客户 EDR 平台 | 下发到对应主机 Agent |
| 4 | EDR Agent | 在主机执行命令 |
| 5 | 主机 | 产生事件 |
| 6 | HIDS / EDR | 监控事件并产生告警 |
| 7 | SIEM | 接收告警 |
| 8 | 客户 EDR 平台 | 回传执行结果给 Veriguard 后端 |
| 9 | Veriguard 后端 | 查 SIEM 关联告警 + 计算验证结果 |

### 3.7 路径对比

| 维度 | 路径 A（OpenAEV Agent 直驱）| 路径 B（第三方 EDR 借力）|
| --- | --- | --- |
| Agent 部署 | 需在主机装 OpenAEV Agent（6 平台 × 2 架构 = 12 二进制）| 无需装新 Agent，复用客户既有 EDR |
| 厂商依赖 | 仅依赖 OpenAEV 开源 | 依赖厂商 API access + remote-script 权限 |
| 通信加密 | 平台自控（TLS）| 走厂商 SaaS / 私部 API |
| 适合场景 | 客户无成熟 EDR / 自主可控诉求 | 客户已部署 EDR 且 API 可用 |
| 国产化兼容 | 需自行编译麒麟 / UOS / 鲲鹏 / 飞腾产物 | 看 EDR 厂商支持范围 |

### 3.8 验证维度

| 维度 | 评估方式 |
| --- | --- |
| **拦截** | 主机攻击行为被阻断（如反弹 shell 失败 / webshell 被删除 / 提权失败）|
| **检测** | SIEM 关联告警匹配（HIDS / EDR）|

## 4. 自定义验证靶机

### 4.1 模块定位

招标自定义验证章节列了 6 类用例，本章不含沙箱样本，覆盖余下 **5 类**。每类用例的靶机环境复用前述 3 个业务模块的拓扑，外加邮件类的特殊路径。

### 4.2 5 类用例 ↔ 靶机环境映射

| # | 自定义用例类型 | 复用拓扑 | Agent | capability | 备注 |
| --- | --- | --- | --- | --- | --- |
| 1 | 构造 web 攻击包 | 边界防护拓扑（第 1 章）| 协作主机 Agent | `http_attack` | 与边界防护完全同路径 |
| 2 | 上传 pcap 流量包 | 流量安全拓扑（第 2 章）| 协作主机 Agent | `pcap_replay` | 与流量安全完全同路径 |
| 3 | 配置执行的命令 | 主机安全拓扑路径 A（第 3.3 节）| OpenAEV Agent | `command_exec` | 主机路径 A 简化版 |
| 4 | 上传可执行文件 + 命令 | 主机安全拓扑路径 A | OpenAEV Agent | `file_drop` + `command_exec` | 多一步 file_drop 落盘 |
| 5 | 配置邮件形式 | **邮件特殊路径**（4.3）| **无 Agent** | — | 平台 SMTP 直发 |

### 4.3 邮件用例的特殊路径

邮件类用例是 5 类自定义里**唯一不需要 Agent** 的 —— 平台后端的 `MailInjector` 直接通过 SmtpProfile 拨号发邮件。

#### 4.3.1 网络拓扑


#### 4.3.2 数据流时序

| 步骤 | 主体 | 动作 |
| --- | --- | --- |
| 1 | 用户 | 在编辑器配置邮件 inject（主题 / 正文 / 附件 / 收件人 teams）+ 选 SMTP profile |
| 2 | Veriguard 后端 | `EmailExecutor` 调 `MailInjector` 按 SmtpProfile 拨号 |
| 3 | SMTP 服务器 | 中转邮件 |
| 4 | 邮件安全网关 | 扫描邮件并产生告警（钓鱼检测 / 附件查毒 / 链接威胁情报）|
| 5 | SIEM | 接收邮件安全网关告警 |
| 6 | 邮件 | 投递到收件人邮箱（若未被退信）|
| 7 | 终端用户 | （人工 / 自动）触发钓鱼链接 / 附件 |
| 8 | Veriguard 后端 | 查 SIEM 关联告警 + 记录投递结果 |

#### 4.3.3 验证维度

| 维度 | 评估方式 |
| --- | --- |
| **拦截** | 邮件被网关退信 / 附件被剥离 / 链接被替换 |
| **检测** | SIEM 关联告警匹配（邮件安全网关）|
| **用户行为** | 收件人是否点击链接 / 打开附件（追踪 pixel / 链接回调）|

## 6. 关键观察

1. **4 个业务模块的 Agent 实际只有 3 类**（去重后）：
   - **协作主机 Agent**（`http_attack` + `pcap_replay` capability，独立 Linux 主机，主动攻击源）
   - **OpenAEV Agent**（`command_exec` + `file_drop` capability，装在被验证主机自身）
   - **第三方 EDR Agent**（不属于 Veriguard 资产，借力客户既有部署）

2. **边界防护 + 流量安全 + 自定义 web / pcap 共用协作主机 Agent** —— 只是 capability 不同（`http_attack` vs `pcap_replay`）。

3. **主机安全 + 自定义命令 / 可执行 + 命令共用 OpenAEV Agent** —— capability 不同（`command_exec` vs `file_drop` + `command_exec`）。

4. **邮件 inject 是 5 类自定义里唯一不需 Agent 的特例** —— 平台自身 SMTP 直发。

5. **派发逻辑全部已实现**（PR-A / PR-B / PR-C / PR-D 全部 merge 到 main），但**协作主机 Agent + OpenAEV Agent 客户端代码**是独立 follow-up 子项目（fork OpenAEV-Platform/agent 加 capability 声明）。客户端代码落地前，PR-C / PR-D 的 dispatch trace 只是"找到 Agent 并记录派发"，真实 HTTP / tcpreplay 执行需待 Agent 项目完成。

## 7. IPv6 单栈环境约束

招标第 1 节明确**所有靶机必须支持 IPv6 单栈环境**：

| 维度 | 招标要求 | 靶机部署影响 |
| --- | --- | --- |
| 地址空间 | 128 位 IPv6 | 靶机网卡必须配 IPv6（v4 可保留作过渡）|
| 终端覆盖 | "对 IPv6 终端的验证特性" | 主机靶机 OS 必须支持 IPv6（Linux ≥ 2.4 / Windows ≥ XP SP2 / MacOS 默认开）|
| 安全基线 | "针对 128 位地址空间特性重构" | 靶机的安全配置（防火墙规则 / 路由）必须 IPv6 适配 |
| 联动 | "与府谷现有 IPv4 安全体系联动" | 靶机环境需双栈（IPv4 + IPv6），同时验证两套防御 |

实施期所有靶机镜像 / docker 镜像必须经过 IPv6 联通性测试。

## 9. 一句话总结

按业务模块推导的 4 类靶机环境（边界 / 流量 / 主机 / 自定义）共用 **3 类 Agent**：协作主机 Agent（攻击源 / 边界 + 流量）、OpenAEV Agent（攻击源 = 被攻击端 / 主机）、第三方 EDR Agent（主机借力）。所有数据流统一遵循"用户编辑 → 平台后端按 capability 选 Agent → Agent 在目标网络发起攻击 → 安全设备产生告警 → SIEM 中转 → 平台查询关联 → 计算验证结果"七步式时序。邮件类用例是唯一不需 Agent 的特例，由平台 SMTP 客户端直接送达。
