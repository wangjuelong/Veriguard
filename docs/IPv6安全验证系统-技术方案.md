# 第一章 方案概述

## 1.1 平台简介

本系统是一套面向 IPv6 网络环境的企业级安全验证平台。平台通过持续、自动化、可量化的攻击模拟手段，对客户既有安全防御体系进行真实有效性验证，让"防御能力可视、可测、可证"成为日常运营的一部分。

不同于传统的渗透测试或漏洞扫描，平台聚焦于"防御是否真的拦得住"与"告警是否真的看得见"两个核心问题，通过持续验证的方式，帮助客户回答以下关键问题：

- 在网络边界部署的检测设备，对新型威胁的覆盖度如何？
- 主机上的安全 Agent，能否及时发现典型攻击行为？
- SOC 告警规则，在真实攻击发生时是否能触发？
- 当一次复合攻击穿越多道防线时，整体防御体系处于什么状态？

本文档面向客户技术决策者、安全运营团队、招标评审专家与基础设施规划工程师，旨在系统性阐述平台的整体技术方案、功能能力与部署要求。

## 1.2 方案核心价值

| 维度 | 价值 |
| --- | --- |
| **完整覆盖** | 完整满足业务需求中规定的全部功能要求，并在多个维度提供企业级配套能力 |
| **可视化呈现** | 攻击编排、执行过程、ATT&CK 战术覆盖等关键能力均以可视化方式呈现，无需技术背景即可理解 |
| **闭环验证** | 从用例编排、攻击执行、防御检测到结果评估，形成完整闭环 |

| **国产化适配** | 原生支持国产化 CPU 与国产化操作系统，IPv6 双栈贯穿全链路 |

## 1.4 技术路线（技术选型）

平台的技术路线以"工业级成熟组件 + 分层解耦架构 + 容器化交付"为核心，所有关键选型均围绕"稳定性、生态成熟度、国产化兼容性、可演进性"四个维度展开。

### 1.4.1 技术栈

平台采用经过工业级验证的成熟技术栈，兼顾性能、稳定性与生态丰富度：

| 层次 | 技术 |
| --- | --- |
| 后端框架 | Spring Boot 3.3.7 / Java 21 |
| 前端框架 | React 19 / TypeScript / Vite |
| 关系数据库 | PostgreSQL 17 |
| 搜索引擎 | Elasticsearch 8 |
| 对象存储 | MinIO |
| 消息队列 | RabbitMQ |
| 任务调度 | Quartz |
| 可视化画布 | ReactFlow |

### 1.4.2 关键架构特性

- **分层解耦**：用户接入层、接入网关层、业务领域层、数据层、外部集成层五层清晰分离，各层职责单一
- **能力接口化**：流量回放设备、主机 Agent、SOC 告警系统、沙箱后端等外部能力均通过标准接口接入，便于按客户既有环境选型
- **演进式数据模型**：基于版本化迁移机制，所有数据结构变更顺序应用、向后兼容
- **容器化交付**：全部组件以容器镜像形式交付，部署一致、运行隔离

### 1.4.3 技术选型说明

每一项技术选择均有明确的取舍依据，确保选型与平台目标对齐：

| 选型维度 | 选定方案 | 选型理由 |
| --- | --- | --- |
| 后端运行时 | Java 21 (LTS) + Spring Boot 3.3.7 | LTS 长期维护版本；Spring 生态在企业级安全产品中占主导地位；与国产化运行环境（毕昇 JDK、龙井 JDK 等）兼容良好 |
| 前端技术 | React 19 + TypeScript + Vite | React 在企业控制台领域生态最广；TypeScript 强类型保障；Vite 构建速度快、HMR 友好 |
| 关系数据库 | PostgreSQL 17 | 强一致性、丰富索引、JSON / 数组等扩展类型支持；可平滑切换至国产化派生版（如人大金仓、瀚高） |
| 搜索引擎 | Elasticsearch 8 | 全文检索、聚合分析、时间序列查询能力齐备；可平滑切换至 OpenSearch |
| 对象存储 | MinIO | 兼容 S3 协议，私有化部署友好；样本、pcap、报告等非结构化数据的标准承载 |
| 消息队列 | RabbitMQ | 异步任务调度、节点执行结果回写场景成熟稳定 |
| 任务调度 | Quartz | 与 Spring Boot 集成度高；支持持久化调度、集群部署、复杂时间表达 |
| 可视化画布 | ReactFlow | 攻击编排链路画布的事实标准选择，支持自定义节点、边、布局算法 |
| 外部能力对接 | 接口化适配层 | 流量回放设备、主机 Agent、SOC、沙箱等异构组件通过标准接口接入，避免硬绑定特定厂商产品 |
| 部署形态 | 容器镜像 | 部署一致性、运行隔离、版本回滚便捷，与客户既有容器编排平台无缝对接 |

### 1.4.4 演进与兼容策略

- **数据库变更**：所有结构变更采用版本化迁移脚本，顺序应用、向后兼容
- **接口演进**：业务接口采用增量演进方式，旧版本接口在过渡期内保留
- **依赖升级**：定期同步上游组件安全补丁；重大版本升级前在测试环境完成兼容性验证

## 1.6 工作范围与项目目标

### 1.6.1 项目目标

本项目以构建一套面向 IPv6 网络环境的企业级安全验证平台为总体目标，使客户具备"持续、自动、可量化"的安全防御真实有效性验证能力：

- **防御可视**：让网络边界、主机端点、SOC 告警三个层面的防御能力变成可观察的运营指标
- **防御可测**：通过标准化的攻击用例与多种验证模式，对防御能力进行端到端的真实测量
- **防御可证**：以验证记录、报告、仪表板形态沉淀证据链，支撑等保合规与内部安全考核

# 第二章 系统架构

本章阐述平台的整体技术架构、部署形态、关键数据流与组件依赖关系，帮助读者建立对平台技术全貌的认知。

## 2.1 设计原则

平台架构在设计阶段确立了以下五项核心原则，贯穿全部模块与组件：

| 原则 | 含义 |
| --- | --- |
| **分层解耦** | 用户接入、API 服务、业务领域、数据存储、外部集成五层清晰分离，各层职责单一、边界明确 |
| **能力接口化** | 流量回放、主机 Agent、SOC 告警、沙箱后端等外部能力通过标准接口接入，便于客户按既有安全设备选型 |
| **演进式数据模型** | 基于版本化数据库迁移机制，所有数据结构变更顺序应用、向后兼容、可追溯 |
| **容器化交付** | 全部组件以容器镜像形式交付，部署一致、运行隔离 |
| **可观测性内置** | 日志、指标、健康检查协同工作，关键业务事件可观测、可告警、可分析 |

## 2.2 逻辑架构

平台从逻辑上分为五个层次。下图展示了各层组件及其交互关系：

![2.2 逻辑架构](img/diagrams/diagram-01.svg)

### 2.2.1 各层职责

**用户接入层** —— 平台对外的入口

- **Web 控制台**：基于现代前端框架构建的单页应用，覆盖全部功能的可视化操作。安全运营人员通过浏览器即可完成日常工作
- **外部系统接入**：面向第三方系统的标准化平台接入能力，便于自动化集成

**接入网关层** —— 请求接入与权限网关

- 按业务领域组织能力，结构清晰
- 提供统一的认证与鉴权能力：先确认"你是谁"，再确认"你能做什么"
- 权限策略集中声明，便于审查与维护

**业务领域层** —— 平台核心能力

| 引擎 | 职责 |
| --- | --- |
| 攻击编排引擎 | 链路定义、节点关系、执行模式策略 |
| 链路任务调度器 | 任务调度，支持节点延迟、重复、条件分支 |
| 期望评估引擎 | 节点级与链路级期望双层评估，结果计算 |
| SOC 关联分析 | 基于客户既有 SIEM 系统进行告警关联查询 |
| 用例匹配引擎 | 动态 filter 匹配，支持实时联动 |
| 沙箱管理服务 | 沙箱实例 CRUD、网络策略配置、执行后环境还原 |

**数据层** —— 多种存储介质各司其职

| 存储 | 承载内容 |
| --- | --- |
| PostgreSQL | 核心关系数据（用户、资产、用例、链路、运行记录等） |
| Elasticsearch | 全文搜索、ATT&CK 矩阵聚合分析、历史结果时间序列 |

# 第六章 核心业务功能

## 6.1 流量安全验证

本节验证客户网络边界部署的检测设备（NTA、IDS 等）对各类流量侧攻击的真实防护效用。

### 6.1.1 模块定位

流量安全验证是平台的核心能力之一。安全运营人员通过本模块可以回答以下关键问题：

- **覆盖度问题**：网络边界部署的检测设备，到底覆盖了哪些资产？哪些资产是检测盲区？
- **稳定性问题**：检测设备是否长期稳定工作？最近一段时间的检出率有无下滑？
- **有效性问题**：对反弹 shell、暴力破解、远控木马等典型攻击，检测设备是否真能拦得住、告得出？

平台通过持续、自动、可量化的流量回放验证，让网络边界防御能力的真实状态变得可视、可测、可证。

### 6.1.2 网络边界资产覆盖度验证

平台对客户网络中的全部资产建立结构化清单，并以可视化矩阵的形式展示每个资产被流量检测设备及其策略覆盖的情况：

- **覆盖缺口高亮**：未被任何设备覆盖、或未被关键策略覆盖的资产以高亮方式显示
- **覆盖度汇总**：按业务系统、网段、资产类型多维度汇总覆盖率

### 6.1.3 流量安全设备稳定性验证

平台支持创建周期性稳定性验证任务，对检测设备的长期工作状态进行趋势化跟踪：

- **任务编排**：设定验证频率（如每日 / 每周 / 自定义周期）、目标设备、用例集
- **稳定性趋势图**：横轴为时间、纵轴为检出率，可视化展示设备性能变化

## 6.3 自定义验证

本节支持用户根据业务需要自定义验证场景、批量导入用例、设计多种类型的自定义用例，并通过动态筛选实现用例集随用例库自动联动。

### 6.3.1 模块定位

通用攻击场景验证解决的是"行业典型威胁"问题，而自定义验证解决的是"客户特定需求"问题。每个客户的业务系统、数据敏感度、合规要求都不同，平台必须支持客户：

- **构建自己的验证场景**：把若干用例按特定顺序组合，模拟真实业务受到的威胁
- **构造特定的用例**：当通用用例库不够用时，自己定制 web 攻击包、流量包、命令、邮件等
- **自动响应用例库变化**：用例库新增的用例如果符合场景条件，自动进入场景

平台为这三个需求分别提供完整支持。

### 6.3.2 自定义场景与执行顺序

平台支持用户自定义验证场景：

- **用例顺序自定义**：用户在可视化画布上手动设定场景内用例的执行顺序
- **基于 ATT&CK 维度自动划分**：场景内用例可按 ATT&CK 战术阶段（侦察、武器化、投递、利用、安装、命令与控制、目标行动）自动归类
- **基于纵深防御维度自动划分**：场景内用例可按纵深防御层级（网络边界、主机端点、应用、数据等）自动归类
- **双维度切换**：用户可在 ATT&CK 视图与纵深防御视图之间一键切换

### 6.3.3 批量导入用例生成场景

平台支持通过批量导入用例的形式快速生成自定义场景：

- 支持结构化数据格式（CSV、JSON 等）批量导入用例
- 导入时可同步设定场景元数据（名称、描述、标签）
- 导入过程支持校验、预览、确认
- 导入失败的条目独立标注，便于修正后重导

### 6.3.4 多种类型的自定义用例

平台支持以下 6 种类型的自定义用例：

| 用例类型 | 用途 | 关键能力 |
| --- | --- | --- |
| 构造 web 攻击包 | 自定义 HTTP/HTTPS 请求作为攻击载荷 | URL、方法、Header、Body、参数、Cookie 自由配置 |
| 上传 pcap 流量包 | 上传客户自有的 pcap 文件用于回放 | 文件上传、文件元数据、回放速率 |
| 上传样本文件 | 上传恶意样本文件（如病毒、木马）用于沙箱执行 | 文件上传、样本类型标注、沙箱选择 |

| 配置邮件形式 | 配置攻击邮件（钓鱼、附件、链接等） | 收件人、主题、正文、附件、内嵌链接 |

### 6.3.5 用例筛选生成验证场景

平台支持用户从全量用例库中通过筛选条件创建自定义验证场景：

- **多维筛选**：按 ATT&CK 攻击模式、杀伤链阶段、攻击类型、攻击载体、标签等多维度组合筛选

## 6.4 攻击编排

本节支持用户在可视化画布上构建多节点攻击链路，模拟真实复合攻击场景，并通过链路化执行验证整体防御体系的有效性。

### 6.4.1 模块定位

真实世界的高级攻击很少是"单点行为"，往往是"侦察 → 投递 → 利用 → 持久化 → 横向移动 → 数据外传"的多步链路。传统的单用例验证无法回答以下问题：

- 当攻击者在前置阶段已经成功时，后续阶段的防御能否补救？
- 不同阶段的检测设备之间，告警是否能正确关联？
- 整个防御纵深体系，在面对一次完整攻击时，最终是"全链路有效"、"部分失效"还是"全链路失效"？

攻击编排模块正是为回答这些问题而设计。安全运营人员通过可视化画布，把多个用例按"链路"形式组合，平台按用户定义的逻辑依次执行各节点，并对整个链路的最终防御效果做综合判定。

### 6.4.2 可视化链路编排

平台提供基于现代前端框架构建的可视化攻击编排画布：

- **拖拽式节点**：从用例库拖拽用例到画布即生成节点
- **可视化边连接**：节点间用鼠标拖拽建立执行依赖
- **画布交互**：支持缩放、平移、自动布局、节点搜索

### 6.4.3 节点用例自定义配置

每个节点对应一个具体用例，用户可在节点级别自定义：

- 用例选择：从用例库挑选具体用例
- 用例参数：覆盖用例的默认参数
- 目标资产：指定本节点的攻击目标
- 预期结果：本节点的验证期望

### 6.4.4 节点执行配置

每个节点支持灵活的执行行为配置：

| 配置项 | 含义 |
| --- | --- |
| **延迟执行** | 节点是否延迟执行，延迟时间（秒级精度） |
| **重复执行** | 节点是否重复执行 |
| **重复次数** | 重复执行的总次数 |
| **重复间隔** | 两次重复之间的时间间隔 |

这些配置便于模拟真实攻击中的"低频试探"、"集中爆破"等行为模式。

### 6.4.5 节点间执行逻辑

节点之间不仅有"先后顺序"，还有基于结果的"条件分支"：

- **执行依赖**：节点 B 必须在节点 A 执行完成后才能执行
- **条件分支**：节点 B 是否执行取决于节点 A 的执行结果（如"A 被拦截才执行 B"、"A 成功才执行 B"）
- **复合条件**：多个父节点状态的与 / 或组合作为子节点执行条件

### 6.4.6 全局与节点级验证参数

平台支持两个层级的验证参数配置：

- **全局参数集**：在链路任务级别配置，应用于全部节点
- **节点级参数**：单个节点可覆盖全局参数，使用独立的参数集
- **参数集复用**：参数集独立维护，多个链路任务可共享同一参数集

### 6.4.7 任务执行模式

平台支持两种链路任务执行模式：

- **拦截后继续**：任一节点被拦截后，链路仍继续执行后续节点
  - 适用于评估纵深防御能力（看后续防御层是否也能拦截）
- **拦截后停止**：任一节点被拦截后，立即停止整条链路
  - 适用于评估单点防御效果（一旦拦下就不必继续验证）

模式可在链路任务创建时选择。

### 6.4.8 SOC 关联告警验证

平台支持使用整个攻击链路验证客户 SOC 关联告警规则的有效性：

- 链路级 SOC 期望配置：定义整条链路应触发哪些关联告警规则
- 多 SOC 连接器适配：支持对接多种 SIEM 系统
- 告警匹配条件：可定义告警规则匹配的字段、时间窗口、关联关系
- 验证结果：明确指示每条关联规则是否在本次任务中触发

### 6.4.9 运行画布展示

链路任务执行过程中，平台通过运行画布实时展示每个节点的拦截和检测结果：

- 节点状态实时着色（成功 / 失败 / 部分失效 / 待评估）
- 拦截层与检测层的结果同时呈现
- 整体链路状态在画布顶部聚合显示
- 节点详情侧栏：点击节点查看详细执行记录、期望对比、关联告警

### 6.4.10 时间轴展示

除画布视图外，平台还提供时间轴视图，从时间维度展示任务执行细节：

- **横轴时间**：从任务开始到结束的完整时间线
- **纵轴节点**：所有节点按链路结构纵向排列
- **节点状态在时间轴上的位置**：直观呈现"何时开始 / 何时结束 / 持续多久"
- **期望与告警事件**：在时间轴上标记期望评估时间点与告警触发时间点

### 6.4.11 三态验证结果

平台对每个链路任务计算最终的整体验证结果：

| 验证结果 | 含义 |
| --- | --- |
| **全链路有效** | 链路上的关键节点均被防御体系拦截或检测到，防御体系整体有效 |
| **部分失效** | 链路上部分节点未被拦截或检测到，防御体系存在局部盲区 |
| **全链路失效** | 链路上的关键节点均未被拦截或检测到，防御体系整体失效 |

## 6.5 沙箱管理

本节提供沙箱环境的统一管理，支持在隔离环境中安全执行真实恶意样本，验证终端防御体系对真实威胁的响应能力。

### 6.5.1 模块定位

部分高威胁样本（如勒索病毒、挖矿病毒、内核驱动型恶意软件）只有真实执行才能触发完整的攻击行为，从而验证客户的终端安全能力。然而真实执行这些样本风险极高，必须在隔离的沙箱环境中进行。

沙箱管理模块解决以下问题：

- **如何安全运行真实恶意样本**：在隔离环境内执行，不影响生产网络与主机
- **如何管理多个沙箱环境**：平台需统一管理多个沙箱实例与各自的网络策略
- **如何避免样本污染环境**：每次执行完毕自动还原，确保下次执行的环境是干净的

### 6.5.2 真实恶意样本执行

平台支持在沙箱环境中执行真实恶意样本，覆盖以下多类样本：

| 样本类型 | 验证场景 |
| --- | --- |
| 勒索病毒样本 | 验证终端 EDR 对加密行为的检测与阻断能力 |
| 挖矿病毒样本 | 验证对持续高 CPU 占用、加密货币挖矿流量的识别能力 |
| 蠕虫病毒样本 | 验证对横向移动行为的检测与阻断能力 |
| 终端恶意驱动加载 | 验证对内核驱动安装、签名校验、Rootkit 行为的检测能力 |
| 终端权限提升 | 验证对各类提权技术的检测能力 |
| 终端系统账号窃取 | 验证对凭据窃取（如 LSASS 内存读取、SAM 文件操作）的检测能力 |

样本执行过程中，平台收集行为指标（进程、文件、网络、注册表操作）与终端安全产品的告警事件，综合评估防御效果。

### 6.5.3 沙箱平台 CRUD 管理

平台提供沙箱实例的完整生命周期管理：

| 操作 | 能力 |

### 6.5.4 网络访问控制策略

平台支持为每个沙箱配置网络访问控制策略，确保样本执行期间产生的网络行为不会逃逸出沙箱：

| 策略维度 | 配置内容 |

| **流量速率** | 流量速率限制，防止 DDoS 类样本影响外部网络 |

策略以可视化界面配置，下发到沙箱后端生效。

- 还原完成后，沙箱状态自动切回"就绪"，可接收下一次任务

## 7.1 用户与权限管理

本节为平台提供完整的身份管理与访问控制能力，确保不同角色的用户在各自职责范围内安全、合规地使用平台。

### 7.1.1 模块定位

安全验证平台承载着大量敏感能力：攻击用例的执行、生产网络资产的访问、SIEM 告警系统的查询、沙箱样本的运行等。任何一项能力被错误的人使用都可能造成严重后果。用户与权限管理模块解决以下问题：

- **谁能登录平台？** —— 身份管理
- **每个用户能做什么？** —— 角色与权限
- **谁能访问哪些资源？** —— 资源级授权

平台采用业界成熟的 RBAC 模型，结合资源级授权机制，实现细粒度的访问控制。

### 7.1.2 用户管理

平台支持完整的用户生命周期管理：

| 能力 | 内容 |
| --- | --- |
| 用户创建 | 邮箱、用户名、姓名、所属团队、初始角色 |
| 用户编辑 | 修改基础信息、所属团队、角色分配 |
| 用户停用 | 临时禁用用户账号，禁用期间无法登录 |
| 用户删除 | 注销用户账号，关联数据按策略保留或匿名化 |
| 密码策略 | 复杂度要求、过期周期、历史密码复用限制 |
| 多因素认证 | 可选启用（如 TOTP） |
| 单点登录 | 支持对接客户既有的统一身份认证系统 |

### 7.1.3 角色管理

平台采用基于角色的访问控制（RBAC）模型：

- **预置角色**：平台预置若干常用角色（如管理员、安全运营、审计员、只读用户等），覆盖典型岗位职责
- **自定义角色**：客户可基于实际组织架构创建自定义角色，灵活组合权限
- **角色权限编辑**：每个角色对应一组权限点（动作 × 资源类型），可视化编辑
- **角色多人共享**：一个角色可分配给多个用户，权限统一变更

### 7.1.4 权限模型

平台权限以"动作 × 资源类型"二维模型表达：

| 维度 | 取值示例 |
| --- | --- |
| 动作 | 创建、读取、更新、删除、执行、授权等 |
| 资源类型 | 攻击链路、用例契约、资产、用户、报告等 |

例如"对攻击链路的执行权限"、"对用例契约的创建权限"等都是独立的权限点。

权限检查在每次接口调用时由平台核心自动完成，业务代码无需重复编写权限校验逻辑。

### 7.1.5 团队管理

团队是平台内更细的协作单元：

- 一个用户可参与多个团队
- 资源可授权给团队整体，便于团队协作
- 团队管理员可在团队内分配权限

### 7.1.6 资源级授权（Grants）

除基于角色的权限外，平台还支持资源级显式授权：

- 可对单个资源（如某条攻击链路）授予特定用户 / 团队访问权限
- 适用于跨角色的临时授权、特定资源的细粒度共享等场景
- 授权动作全程记入审计日志，可追溯

## 7.2 资产管理

本节为平台提供资产清单与资产组管理能力，是流量验证、主机验证、攻击编排等核心功能的目标基础。

### 7.2.1 模块定位

安全验证活动必须有明确的"目标"。无论是验证流量检测设备对哪些资产生效，还是在哪些主机上模拟攻击行为，平台都需要先有一份准确的资产清单。资产管理模块解决以下问题：

- **平台知道客户网络里有哪些资产吗？** —— 资产清单维护
- **怎么按业务、操作系统等维度组织资产？** —— 标签与分组
- **新增 / 下线资产时，验证场景如何自动跟随？** —— 动态资产组

### 7.2.2 资产清单

平台支持以下资产类型：

| 资产类型 | 用途 | 关键属性 |
| --- | --- | --- |
| 主机资产（Endpoint） | 主机验证、攻击编排目标 | IP（IPv4 / IPv6）、主机名、操作系统、Agent 状态 |
| 通用资产（Asset） | 各类网络与基础设施目标 | 名称、类型、IP、标签 |

### 7.2.3 资产清单管理

| 搜索筛选 | 按 IP、主机名、标签、操作系统等多维筛选 |
| 资产标签 | 自由打标，便于跨维度组织资产 |

### 7.2.4 资产组

资产组是资产的逻辑集合，便于在验证场景中批量选择目标。平台支持两种成员模式：

**静态成员**

- 用户手动指定哪些资产属于该组
- 成员变更需手动维护
- 适用于稳定的资产集合

- 平台按筛选条件实时匹配资产作为成员
- 资产清单更新时，动态组成员自动跟随更新
- 适用于规模化场景与持续变化的环境

## 7.3 用例库管理

本节管理平台的全部攻击验证用例，是流量验证、主机验证、自定义验证、攻击编排等模块的内容基础。

用例是平台一切验证活动的"原材料"。一个完整、准确、最新的用例库决定了平台能验证什么、验证得多准。用例库管理模块解决以下问题：

- **怎么组织成千上万的用例？** —— 用例分类与索引

### 7.3.2 用例契约

每个用例在平台中表达为一个"用例契约"，描述用例的全部元数据与执行要求：

| 元数据维度 | 内容 |
| --- | --- |
| 基础信息 | 名称、描述、标签、多语言标签 |
| 攻击属性 | 所属攻击类型、攻击载体（流量 / 主机 / 邮件 / 沙箱）、可执行平台 |

| 载荷类型 | 用途 |

### 7.3.4 ATT&CK 攻击模式库

平台内置完整的 ATT&CK 攻击模式库：

- **战术（Tactic）**：覆盖侦察、武器化、投递、利用、安装、命令与控制、目标行动等
- **技术（Technique）**：每个战术下的具体技术
- **子技术（Sub-Technique）**：技术的进一步细化
- **平台关联**：每个技术对应的目标平台（Windows / Linux / macOS / 容器等）

用例通过关联 ATT&CK 攻击模式，让验证活动与行业标准对齐，便于评估防御覆盖度。

### 7.3.5 杀伤链阶段

## 7.5 审计日志

本节为平台提供基础的操作审计能力，记录业务实体的变更事件与操作主体，便于事后追溯。

### 7.5.1 模块定位

安全验证平台承载着大量敏感操作：攻击用例执行、资产清单变更、用户权限调整、SOC 连接器配置等。任何一次操作都可能影响平台后续的验证活动与客户网络安全。审计日志模块解决以下问题：

- **谁在什么时间做了什么？** —— 操作记录
- **业务实体被谁修改了？什么时候？** —— 资源变更追溯

### 7.5.2 业务实体变更审计

平台对核心业务实体的创建、修改、删除事件进行结构化记录：

| 审计内容 | 含义 |
| --- | --- |
| 创建时间 | 实体首次创建的时间戳 |
| 最后更新时间 | 实体最近一次修改的时间戳 |
| 操作主体 | 触发变更的用户标识 |
| 实体类型 | 被变更的资源类型 |
| 实体标识 | 被变更资源的唯一标识 |

审计数据持久化至关系数据库，便于长期保留与追溯。

### 7.5.3 审计写入机制

审计数据由平台核心自动写入，无需业务代码显式调用：

- 业务实体生命周期事件（创建 / 更新 / 删除）由平台统一拦截
- 审计字段（时间、操作主体）自动收集并附加到实体
- 业务代码无需感知审计逻辑，确保审计覆盖的完整性

### 7.5.4 与日志体系协同

审计数据除持久化在数据库外，也通过平台日志体系输出：

- 关键操作以结构化日志形式记入应用日志
- 日志可对接客户既有的日志平台（如 ELK、Graylog、Loki 等）做集中归集与长期保留
- 详见 [§ 5 可观测性](#第五章-可观测性)

## 7.6 任务历史与验证记录

本节为平台提供验证任务的全生命周期记录与查询能力，让每一次验证活动的过程与结果都可查、可回溯。

### 7.6.1 模块定位

安全验证是一个长期、持续的活动。一次验证任务执行完成、看到验证结果之后，并不是工作的终点 —— 客户还需要回答以下问题：

- 过去执行了哪些任务？分别对应什么场景？
- 某次任务的具体节点是哪个出了问题？
- 任务执行的完整时间线、关联告警、日志在哪里查？

任务历史与验证记录模块为这些问题提供完整支持。

### 7.6.2 任务列表

平台维护全部历史验证任务的完整列表：

| 列表维度 | 内容 |
| --- | --- |
| 任务基础信息 | 任务名称、关联场景 / 链路、创建人 |
| 执行时间 | 创建时间、开始时间、结束时间、耗时 |
| 任务状态 | 待开始、运行中、已完成、已取消、执行失败 |
| 验证结果 | 全链路有效 / 部分失效 / 全链路失效 |

任务列表支持多维筛选、排序、关键字搜索，便于快速定位历史任务。

### 7.6.3 任务详情

点击单个任务进入详情页，可查看完整的执行记录：

- **任务基础信息**：执行人、任务参数、所用场景版本
- **任务时间线**：从启动到结束的完整时间轴
- **节点执行明细**：每个节点的执行状态、用例参数、实际结果
- **验证结果**：节点级与链路级验证结果
- **关联告警**：本次任务在 SIEM 中触发的关联告警
- **执行日志**：详细的执行日志记录

## 8.4 验收流程

验收按"自检 → 联合测试 → 阶段确认 → 终验签字"四步流程开展：

1. **实施方自检**：实施方完成全部交付物自测，输出《自测报告》
2. **联合 UAT 测试**：客户与实施方按《验收测试用例集》逐项联合测试，记录测试结果
3. **问题修复与回归**：对 UAT 阶段发现的问题在约定时限内修复并回归测试
4. **终验签字**：双方共同确认所有验收指标达成，签署《项目验收单》并进入质保期

验收过程中发现的问题按严重程度分级处理：

| 等级 | 影响 | 处置要求 |
| --- | --- | --- |
| **致命 / 阻塞** | 核心功能不可用、数据丢失 | 必须修复后才能验收 |
| **严重** | 主要功能受限但有规避方法 | 终验前修复 |
| **一般** | 次要功能或体验问题 | 质保期内修复 |
| **轻微** | 文案、样式等细节问题 | 记录跟踪，分批处理 |
