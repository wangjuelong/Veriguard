# 一、紧急澄清事项（启动第 1 周内必答）

## 1.1 安全运营平台对外接口规范

**关注点**：网络边界覆盖度、流量安全验证、主机安全验证、攻击编排链路四大核心场景，最终都需要从甲方安全运营平台读取关联告警以计算「是否被检测到」。接口若不可用，验证结果中的「检测维度」判定整套失效。

**待澄清**：

- 接口形态：REST API / gRPC / Kafka 订阅 / syslog 转发？
- 认证方式：API Key / OAuth 2.0 / mTLS / 无认证？
- 六维查询字段映射：资产 IP、时间窗、严重度、规则 ID、规则类别、关键字 六维 ↔ 平台实际字段名
- 是否同时支持 IPv6 和 IPv4 双栈调用

**项目方倾向答案**：REST API + Bearer Token + 双栈支持。

**备选方案**：若不开放查询 API，改为 syslog 转发 + 本平台自建中转网关 + Elasticsearch 落盘 + 自实现查询。增加约 2 周工作量但不阻塞主线。

**风险**：第三方厂商技术支持响应周期可能拖至 2-4 周。建议第 1 周内甲方提需求，项目方同步启动备选方案。

---

## 1.2 沙箱系统对接

**关注点**：真实恶意样本（勒索病毒 / 挖矿 / 蠕虫 / 木马驱动等八类）执行模块依赖独立沙箱系统提供 API。该接口不可用则沙箱模块全部功能（八类样本执行、自动还原、网络访问控制）无法落地。

**待澄清**：

- 沙箱系统的版本号与 API 文档
- 任务上传、状态查询、报告获取、Snapshot 自动回滚 4 类端点
- 网络访问控制策略下发的数据结构（白名单 / 黑名单 / 协议 / 速率）
- 单节点并发样本数上限
- IPv6 支持情况

**项目方倾向答案**：直接与沙箱负责人对接索取最新文档，明确版本号并写入项目实施文档。

**风险**：沙箱自身的 IPv6 支持和网络策略接口可能尚未实现完整。建议第 1 周内完成评审会议 + 锁定版本号。

---

## 1.3 靶机部署细则

**关注点**：招标要求「多区域部署靶机」，但未列具体数量、区域、操作系统、IPv6 配置和维护责任方。这是项目实施的物理基础——没有靶机就无法进行验证。

**待澄清**：

| 维度 | 待澄清内容 |
| --- | --- |
| **数量** | 靶机总数及在 DMZ / 业务区 / 生产网 / OT 网 的分布 |
| **操作系统** | 国产化清单（麒麟 / UOS / 鲲鹏 / 飞腾）涵盖范围 |
| **IPv6 配置** | 双栈或纯 IPv6 |
| **快照维护** | 镜像 / Snapshot 由甲方维护、项目方维护，还是共同维护 |
| **区域定义** | 是否包含生产 OT 网（SCADA / DCS / PLC） |

**项目方倾向答案**：双栈靶机 30 台（IT 12 台 + DMZ 8 台 + 生产 OT 10 台），各区代表性操作系统覆盖。Snapshot 由项目方主维护，甲方审批。

**风险**：

- **生产 OT 网风险极高**：DCS / SCADA / PLC 不允许直接攻击，需建并行影子环境
- **国产化清单不明**：每个变种需要独立 Agent 编译

**时限**：第 1 周现场勘察；第 2 周完成《靶机部署规范》文档；第 3 周开始靶机准备。

---

## 1.4 客户既有 EDR 现状与 API 可用性

**关注点**：决定主机侧验证走「路径 A：自有 Agent 直驱」还是「路径 B：第三方 EDR 借力」。若客户已部署成熟 EDR 且 API 可用，则路径 B 节省约 3 周开发；否则启动独立 Agent 子项目。

**待澄清**：

- 已部署 EDR 厂商清单（Tanium / CrowdStrike / SentinelOne / Cortex XDR / 国产 EDR 等）
- 各厂商 API 是否对客户开放
- remote-script / RTR / 远程命令执行权限是否可申请
- 既有 HIDS Agent 装机率

**项目方倾向答案**：先调查实际部署情况，路径 A 优先（API 可用时成本最低），fallback 走独立 Agent 部署。

**风险**：路径 A 依赖厂商 SaaS 配额；路径 B 增加多平台多架构编译矩阵；混合方案运维难度大。

---

## 1.5 协作主机 Agent 部署位置与网络资源

**关注点**：流量验证和邮件钓鱼演练已实现派发逻辑，但协作主机 Agent 在客户网内的部署位置、网络权限、数量、IPv6 配置尚未确定。没有这些前置条件，无法发起真实流量。

**待澄清**：

- 数量：1 台共用、每区域 1 台、还是每业务网段 1 台
- 位置：DMZ / 内网管理区 / 业务区 / 镜像口接入点
- 权限：root 直接给，还是最小权限（`CAP_NET_RAW + CAP_NET_ADMIN`）
- 网卡：单网卡 / 多网卡 trunk 接入
- IPv6 配置：必须双栈

**项目方倾向答案**：每区域 1 台（DMZ + 业务区 + 生产 OT 影子各 1 台共 3 台），最小权限模型，多网卡 trunk 接入业务网段。

**风险**：协作主机交付周期不定（物理机 1-2 周，虚拟机工单 ≤ 1 周）。

**时限**：第 1 周现场勘察决定数量与位置；第 1 周末提交《协作主机部署需求文档》；第 2-3 周交付 + 联通性测试。

---

## 1.6 靶机白名单与生产边界

**关注点**：主机攻击包含病毒落盘、提权、持久化等破坏性操作。项目方对生产网攻击造成的直接经济损失承担法律责任，必须有强制机制保证只攻击靶机、不攻击生产机。

**待澄清**：

- 哪些主机可作靶机（IP 段 / 主机名 / 资产组）
- 生产机不可作靶机的硬约束如何技术实现（白名单 vs 黑名单）
- 误执行的回滚 + 应急流程
- 安全委员会对验证 Agent 干扰评估的审批流程

**项目方倾向答案**：白名单 + 任务执行前强制校验 + 安全委员会预审；生产 OT 网额外建立影子环境。

**风险**：

- 安全委员会审批是关键节点，可能延期 2-4 周
- 误执行的应急流程：≤ 1 小时内完成回滚 + 安全事故报告

**时限**：第 1 周提交《验证 Agent 干扰评估文档》初稿给甲方安全委员会；第 2 周实现白名单数据模型；第 3 周安委会终审。

---

## 1.7 集团 IPv6 资产管理平台对接

**关注点**：招标要求与集团 IPv6 资产清单联动。若资产清单不准确，则验证目标对不上，导致网络边界覆盖度验证无法完整执行。

**待澄清**：

- 集团资产管理平台的对外接口（REST / Kafka / 文件同步）
- 数据上报格式与频率
- 认证方式

**项目方倾向答案**：REST API 主选；若不可得，每日定时拉取 Excel / CSV 文件。

**风险**：集团 IT 跨部门协调可能需要 2-3 周。

# 二、重要澄清事项（启动后 2-4 周补齐）

## 集成对接类

### 2.1 安全设备清单与策略导出格式

**关注点**：覆盖度验证需要把「用例 × 策略 × 资产」做矩阵展示。若策略名称对不上，则矩阵为空。

**待澄清**：

- WAF / 防火墙 / IPS / IDS / HIDS 已部署设备清单（厂商、型号、版本）
- 各设备策略导出格式（厂商 API / 配置文件 / Excel 报表）
- 策略命名规范

**项目方倾向答案**：API 优先，无 API 时走 Excel 模板 + 定期同步。

---

### 2.2 测试邮箱 SMTP 服务器配置

**关注点**：邮件钓鱼演练已完成 SMTP Profile 管理界面，但需要甲方填实参数才能跑实战。

**待澄清**：

- 测试用 SMTP 服务器 IP / 端口 / 凭据
- 发件白名单（哪些 `from_alias` 允许）
- 日发送量上限
- 是否允许使用客户真实邮件域名

**项目方倾向答案**：独立测试 SMTP 服务器，避免污染生产邮件流量。

**时限**：第 2 周拿到 SMTP 参数 → 完成端到端测试。

---

### 2.3 恶意样本来源与合规手续

**关注点**：沙箱模块要求八类真实恶意样本。样本「持有」涉及《网络安全法》第 27 条和《刑法》285-287 条，需明确合规边界。

**待澄清**：

- 样本提供责任方（甲方安全研究院 / 项目方自备 / 第三方采购）
- 样本进口、持有的合规手续
- 八类样本各家族优先级（勒索病毒优先 WannaCry / GandCrab / LockBit 哪几个）
- 样本运输与销毁流程

**项目方倾向答案**：甲方安全研究院提供（资质完备，合规风险低）。

**时限**：第 2 周明确责任方；第 3-4 周完成八类样本入库（每类至少 5 个家族变种）。

---

## 用例数量与内容类

### 2.4 边界攻击用例 12 类配比

**关注点**：招标要求边界攻击 12 类、用例总数 ≥ 500 条（或攻击类型 ≥ 10 类）。配比方式影响最终验收呈现。

**待澄清**：

- 12 类是否每类都需配，还是只需 ≥ 10 类满足
- 每类配比下限（SQL 注入、XSS 主流类 vs CSRF、SSTI 偏冷门类）
- 数据集来源是否接受开源（Nuclei templates、OWASP CRS、PayloadsAllTheThings）

**项目方倾向答案**：主流类高配比（SQL 注入 100 / XSS 80 / 命令执行 80 / 其他 9 类 ≥ 30），总数 ≥ 1000 条。已超额准备。

---

### 2.5 八大软件类高危漏洞 PoC 配比

**关注点**：招标列了八大软件类（主流 web 组件 / 安全产品 / 应用程序 / 国产商业软件 / CMS / 网络产品 / 操作系统 / 数据库），每类下面具体支持哪几家厂商不明。

**待澄清**：

- 国产数据库选哪几款（达梦 / 人大金仓 / 巨杉 / GBase / TiDB）
- 国产操作系统选哪几款（麒麟 / UOS / OpenEuler / 中科方德 / 凝思）
- 国产商业软件（用友 / 金蝶 / 通达 / 泛微 / 蓝凌 / 致远）
- 每类至少几个 CVE PoC

**项目方倾向答案**：覆盖客户实际可能用到的国产软件，每类 ≥ 10 个 CVE PoC。

---

### 2.6 三万攻击组合执行窗口

**关注点**：动态生成的三万个攻击组合若全量并发跑，可能压垮防护设备或业务网络。

**待澄清**：

- 允许的执行时间窗口（非业务高峰 / 凌晨 / 周末）
- 单次任务可跑的最大组合数
- 并发上限
- 是否分批跑

**项目方倾向答案**：常态业务低峰 + 限流到设备容忍范围；大批量验证（验收演示）走凌晨窗口。

---

### 2.7 攻击组合分级标准

**关注点**：三万攻击组合的「不能防护问题」需要分级。

**待澄清**：

- 是否接受 4 级默认（紧急 / 严重 / 一般 / 提示）
- 分级判定算法（按 CVSS / 资产关键性 / 防护设备覆盖率）
- 是否需要自定义分级权重

**项目方倾向答案**：4 级默认 + 高级配置项允许客户自定义权重。

---

### 2.8 流量用例 IPv6 单栈覆盖度

**关注点**：流量攻击用例 11 类，每条是否要 IPv4 + IPv6 双版本。

**待澄清**：

- 每条流量用例是否要 IPv4 + IPv6 双版本
- 是否接受 Suricata-Verify 等开源 pcap 作为来源
- 真实攻击 pcap 优先源（如 malware-traffic-analysis.net）

**项目方倾向答案**：IPv6 主、IPv4 备（与招标本意一致）。

---

### 2.9 主机自动还原标准

**关注点**：12 类主机攻击各自的「自动还原」标准不一，需要逐类明确。

**待澄清**：

- 每类的还原必做项（删落盘文件 / 回滚注册表 / 杀进程 / 清持久化）
- 还原失败的兜底处理

**项目方倾向答案**：每类用例契约新增 `rollback_steps` 字段，由 Agent 端解析执行。参考还原矩阵：

| 攻击类型 | 还原必做项 |
| --- | --- |
| 反弹 shell | 杀进程 + 关闭端口 + 删脚本 |
| webshell 上传落盘 | 删文件 + 清访问日志 |
| 系统提权 | 撤销提权 token / 重启服务 |
| 主机持久化 | 删 cron / 服务 / 启动项 / 注册表 Run |
| 病毒样本落盘 | 删文件 + 清隔离区 |

---

## 验收判定规则类

### 2.10 稳定性指标计算公式

**关注点**：边界设备稳定性、流量设备稳定性两条标星条款，但「稳定性」无统一定义。

**待澄清**：

- 稳定性 = 命中率（命中次数 / 重复次数），还是其他指标
- N 次重复执行的 N 默认值（10 / 100 / 1000）
- 趋势图聚合粒度（按任务 / 按天 / 按周）

**项目方倾向答案**：命中率 = 命中次数 / N，N 默认 10，客户可配。

---

### 2.11 攻击编排 SOC 六维匹配字段

**关注点**：攻击编排链路与 SOC 告警关联匹配，是标星条款强制项。

**待澄清**：

- 六维（资产 IP / 时间窗 / 严重度 / 规则 ID / 规则类别 / 关键字）↔ 安全运营平台实际字段名映射表
- 是否允许节点级覆盖全局参数
- 规则 ID 是平台 ID 还是设备 ID

**项目方倾向答案**：六维通用模型 + 安全运营平台适配层做字段翻译。

**依赖**：本项依赖紧急澄清 1.1（安全运营平台接口）完成后才能定。

---

### 2.12 攻击编排三态结果判定

**关注点**：链路任务「全链路有效 / 失效 / 部分失效」三态的具体阈值不明。

**待澄清**：

- 「全链路有效」= 所有节点全部失败，还是关键路径节点全部失败
- 「全链路失效」= 所有节点全部成功
- 「部分失效」阈值（≥ 1 节点失败即算 vs ≥ 50% 节点失败）
- 是否允许自定义判定规则

**项目方倾向答案**：默认按所有节点至少一维（拦截或检测）成功来判定，并提供节点级权重配置选项。

---

### 2.13 沙箱网络访问控制四维策略

**关注点**：招标「沙箱平台网络访问控制策略」为强制截图项，否则视为无效。

**待澄清**：

| 维度 | 内容 |
| --- | --- |
| 白名单 | 允许出网的 IP / 域名 |
| 黑名单 | 禁止访问的 IP / 域名 |
| 协议 | HTTP / DNS / ICMP 各自允许 |
| 速率 | 单连接带宽 / 总带宽限制 |
| 预设策略 | 完全隔离 / 仅 DNS / 仅外网 / 受控外网 |

**项目方倾向答案**：四维完整 + 4 个预设策略，由沙箱适配层下发。

---

### 2.14 动态用例集联动实时性

**关注点**：用例库变更时，自定义验证场景内容是否自动实时更新。

**待澄清**：

- 「自动进入」的实时性要求（实时 / 1 分钟 / 1 小时 / 次日批量）
- 是否需要 UI 提示用户场景内容已变更

**项目方倾向答案**：分钟级（定时任务），平衡实时性与性能；UI 提供徽标提示。

## 数据 / 报告 / 截图 / 合规类

### 2.15 截图素材的合规与脱敏要求

**关注点**：标星条款均要求功能截图。截图含何字段、是否脱敏、归档周期都不明。

### 2.16 验证报告生成与对外披露

**关注点**：交付文档清单包含八份 Word 报告。每份的频率、接收人、格式不明。另含发明专利 2 项 + 核心期刊论文 2 篇。

**待澄清**：

- 报告生成频率（每次验证 / 每月 / 每季度）
- 报告接收人范围（甲方安全委员会 / 集团 / 主管单位）
- 格式约定（Word 模板 / PDF / 含原始数据附件）
- 专利与论文的合作署名与撰写责任

**项目方倾向答案**：实时大屏 + 月报 Word + 节点性 Word 三层组合。

# 三、一般澄清事项（试运行期补齐）

> 以下项不阻塞主线研发，可在试运行期通过 weekly review 补齐。

### 3.1 用户角色与权限模型

四个默认角色（管理员 / 操作员 / 审计员 / 只读）已实现 RBAC 权限矩阵。若客户要求集成集团 IAM 角色映射，在试运行期补适配。

### 3.2 平台可用性与容灾

默认承诺 99.9% SLA + 单机部署。如需 HA 主备，追加硬件后实施。PostgreSQL 每日全备 + Elasticsearch snapshot；RTO 4 小时 / RPO 24 小时默认。

### 3.3 升级维护窗口

默认每月第 1 周末 22:00-02:00 维护窗口。升级走容器滚动重启 + 数据库迁移；DB 备份 + 镜像版本锁定支持回滚。

### 3.4 数据导入导出格式

默认 Excel 模板（用例批量导入）+ JSON 导出（链路 / 配置）。支持 STIX 2.1 + ATT&CK 标准格式，与 MITRE 生态互通。

### 3.5 报告输出频率与渠道

默认 dashboard 实时 + 月报 Word + 季报 Word + 突发事件即时报告。邮件订阅每月 1 号自动发送。

### 3.6 性能基线承诺

| 指标 | 默认承诺 |
| --- | --- |
| 平台并发任务数 | ≥ 100 |
| 三万攻击组合执行时间 | ≤ 4 小时（限流 50 并发） |
| 单次 inject 的 P95 延迟 | ≤ 5 秒 |
| Agent 心跳间隔 | 30 秒 |
| Agent 结果上报间隔 | 1 分钟 |

### 3.7 IPv6 单栈程度

默认平台内部纯 IPv6 + 外部对接（安全运营平台 / 沙箱 / SMTP / 集团 IPv6 平台）双栈。DNS / NTP / 镜像源用 IPv6 链路。

### 3.8 培训教材的产权与延续

教材项目方主编 + 甲方共享版权（甲方可二次分发）。培训后 1 年内 ≤ 2 次复训免费。线上培训含录像存档。

### 3.9 攻击特征库更新机制

默认每月 1 次特征库更新（用例库 / 漏洞库 / 攻击样本库同步）+ 紧急 CVE 24 小时内补充。离线更新方案：项目方提供升级包，甲方人工导入。

### 3.10 多区域部署网络拓扑细节

默认每个「区域」独立 VLAN + IPv6 段。跨区域流量经客户既有防火墙。协作主机 Agent 每区一台独立实例。具体 IP 段在靶机部署细则中一并对齐。
