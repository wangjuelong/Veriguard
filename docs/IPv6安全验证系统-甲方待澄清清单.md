### 1.1 蓝盾 NxSOC 对外接口规范

| 项 | 内容 |
| --- | --- |
| 触发条款 | 招标 §3 / §4 / §7（SOC 关联告警）+ GAP §3.1 / §4.1 / §5.1 / §7.5 |
| 待澄清内容 | （a）NxSOC API 文档（REST / gRPC / Kafka topic？）；（b）认证方式（API Key / OAuth / mTLS）；（c）查询字段映射（资产 IP / 时间窗 / 严重度 / 规则 ID / 规则类别 / 关键字 6 维 ↔ NxSOC 实际字段名）；（d）是否同时支持 IPv6 + IPv4 双栈调用 |
| 影响 | 阻塞 `nxsoc_adapter` 子模块（GAP §3 跨模块新增 #6）；阻塞验证结果计算（拦截维度 + 检测维度）的回环 |
| 备选方案 | 若 NxSOC 无 API → 退而以 syslog 网关 + 自建 SIEM 中转方案承接（增加 2 周工作量） |
| 建议处理方式 | 启动会议时甲方提供文档；如客户内部协调周期长，**研发同步启动 syslog 备选适配** |

**问题本质**：NxSOC 是客户的安全运营平台，平台 4 个核心模块（覆盖度验证 / 流量验证 / 主机验证 / 攻击编排）的"验证结果计算"最后一步都需要查 NxSOC 关联告警。如果接口不可用 → 检测维度判定整套失效 → 招标 ≥ 50% 验收指标失败。

**可能答案 / 选项**：

| 选项 | 描述 | 优势 | 劣势 |
| --- | --- | --- | --- |
| A | NxSOC 提供完整 REST API + 6 维查询 | ✅ 理想情况，1 周内 `nxsoc_adapter` 开发 | 文档可能不全 / 字段语义需逐条验证 |
| B | NxSOC 只提供 syslog 转发，无查询 API | 实现简单 | 平台需自建 SIEM 中转网关 + 落 ES + 自实现查询（+ 2 周） |
| C | NxSOC 提供 Kafka topic 订阅 | 实时性好 | 需平台改造为流式消费 |
| D | NxSOC 不提供任何对外接口 | — | 需走"日志文件定时拉取"等降级方案，且实时性差 |

**项目方倾向**：选项 A。若不可得 → 选项 B（syslog 备选已在 GAP 风险点提前预案）。

**风险与权衡**：
- 🔥 **高风险**：客户内部协调蓝盾文档可能拖 2-4 周（蓝盾是第三方厂商）
- 缓解：第 1 周同时提需求给客户 + 直接联系蓝盾原厂技术支持

**评估证据**：
- NxSOC 产品文档 PDF
- 6 维字段在 NxSOC 中的实际列名
- 网络可达性测试（平台所在网段 → NxSOC IP:Port，含 IPv6）

**后续动作**：
- [ ] 第 1 周：项目方 PM 提交《NxSOC 接口需求清单》给甲方
- [ ] 第 1 周：项目方架构师同步评估 syslog 备选方案
- [ ] 第 2 周：如未拿到 API 文档，启动 syslog 备选实施

### 1.4 客户既有 EDR 现状与 API 可用性

| 项 | 内容 |
| --- | --- |
| 触发条款 | 招标 §5 主机安全验证（HIDS 等）+ 靶机环境第 3 章「路径 A vs 路径 B」 |
| 待澄清内容 | （a）已部署的 EDR 厂商清单（Tanium / CrowdStrike / SentinelOne / Cortex XDR / 国产 EDR 如奇安信 / 深信服 / 启明星辰）；（b）EDR 厂商 API 是否对客户开放；（c）remote-script / RTR / 远程命令执行权限是否可申请；（d）有无既有 HIDS Agent 已装机率 |
| 影响 | 决定主机侧走「路径 A：OpenAEV Agent 直驱」还是「路径 B：第三方 EDR 借力」；如全走路径 A，则需排期 `veriguard-agent` 客户端项目（≥ 3 周） |
| 建议处理方式 | 甲方提供既有 EDR 资产清单 + 厂商联系方式，项目方评估 API 借力可行性 |

**问题本质**：决定主机侧验证走"路径 A：OpenAEV Agent 直驱"还是"路径 B：第三方 EDR 借力"。如果客户已部署成熟 EDR 且 API 可用 → 路径 B 省 3 周开发；否则 → 路径 A 启动独立 Agent 子项目。

**可能答案 / 选项**：

| 选项 | 客户场景 | 项目方采取 |
| --- | --- | --- |
| A | 客户已部署 Tanium / CrowdStrike 且 API 可调用 | 路径 B：复用上游 OpenAEV executor 适配（已实现）|
| B | 客户部署了国产 EDR（奇安信天擎 / 深信服 / 启明星辰）但 API 受限 | 路径 A + B 混合：API 受限部分走路径 A |
| C | 客户无 EDR 或 EDR API 不可用 | 路径 A：装 OpenAEV Agent（fork 项目）|
| D | 客户禁止平台调 EDR API（合规约束） | 路径 A 强制 |

**项目方倾向**：先调查实际部署 → 选项 A 优先（成本最低）；fallback 选项 C。

**风险与权衡**：
- 路径 B 依赖厂商 SaaS / API access 配额
- 路径 A 增加 6 平台 × 2 架构 = 12 二进制编译 + 信创变种
- 混合方案最复杂（运维难度大）

**评估证据**：
- 客户 EDR 资产清单（厂商 / 版本 / 装机率）
- EDR API 凭据获取流程时间窗
- 客户 EDR 厂商现场支持响应能力

**后续动作**：
- [ ] 第 1 周：甲方 IT 提供已部署 EDR 厂商清单
- [ ] 第 1 周：评估每个厂商 API 可用性
- [ ] 第 2 周：决定路径 A / B / 混合，开 ADR

### 1.5 协作主机 Agent 部署位置与网络资源

| 项 | 内容 |
| --- | --- |
| 触发条款 | PR-C web_attack + PR-D pcap_replay 已落地派发逻辑，但未确定 Agent 在客户网内的部署位置 + 网络对接 |
| 待澄清内容 | （a）协作主机 Linux 服务器若干台（每个验证目标网段至少 1 台）；（b）需 root 或 `CAP_NET_RAW` + `CAP_NET_ADMIN`（tcpreplay 必需）；（c）需访问业务 Web 应用（http_attack）的内网或 DMZ 位置；（d）镜像口 / VLAN trunk 接入位置（pcap_replay）；（e）IPv6 单栈对接的网卡配置 |
| 影响 | 阻塞 PR-C / PR-D 的真实执行链路落地（当前 dispatch trace 只能找到 Agent，没有 Agent 实际能跑）|
| 建议处理方式 | 现场勘察 + 网络变更工单；甲方网管 + 项目方网络工程师对齐 |

**问题本质**：PR-C / PR-D 已落地派发逻辑，但协作主机 Agent 物理放在客户哪里、需要什么网络权限、几台、IPv6 配置 都未定。没这些就没法做"真实 HTTP / pcap 流量发起"。

**可能答案 / 选项**：

| 维度 | 选项 |
| --- | --- |
| 数量 | 1 台共用 / 每区域 1 台 / 每业务网段 1 台 |
| 位置 | DMZ / 内网管理区 / 业务区 / 镜像口接入点 |
| 权限 | root 直接给 / `CAP_NET_RAW + CAP_NET_ADMIN`（最小权限） |
| 网卡 | 单网卡 / 多网卡（每个区域一个）|
| IPv6 | 必须 IPv6 双栈 |

**项目方倾向**：每区域 1 台（DMZ + 业务区 + 生产 OT 影子各 1 台共 3 台），最小权限 `CAP_NET_RAW + CAP_NET_ADMIN`，多网卡 trunk 接入业务网段。

**风险与权衡**：
- 协作主机 Agent 本身要装 OS + curl + tcpreplay + Rust runtime
- 若客户提供物理机：等运维交付（可能 1-2 周）
- 若用虚拟机：等 VMware / OpenStack 工单

**评估证据**：
- 协作主机硬件配置要求（2C 4G 50G SSD 起步）
- 镜像口配置 / VLAN trunk 配置可行性

**后续动作**：
- [ ] 第 1 周：现场勘察 + 决定数量 / 位置
- [ ] 第 1 周：提交《协作主机 Agent 部署需求文档》给甲方运维
- [ ] 第 2-3 周：协作主机交付 + 联通性测试

### 1.6 靶机白名单与生产边界

| 项 | 内容 |
| --- | --- |
| 触发条款 | 招标 §5 主机攻击包含病毒落盘 / 提权 / 持久化 + 自动还原；GAP §5.1 靶机白名单机制；招标 §8 真实恶意样本 |
| 待澄清内容 | （a）哪些主机可作靶机（IP 段 / 主机名 / 资产组）；（b）生产机绝对不可作靶机的硬约束如何技术实现（白名单 vs 黑名单）；（c）误执行的回滚 + 应急流程；（d）攻击编排任务对生产网影响评估文档（甲方批准） |
| 影响 | 直接影响"自动还原"机制设计 + 误执行风险评估 + 项目方法律责任边界 |
| 建议处理方式 | 第 1 周提交《验证 Agent 干扰评估文档》给甲方安委会批准；建立靶机白名单表与执行前强制校验 |

**问题本质**：项目方对生产网攻击造成的直接经济损失负法律责任。必须有强制机制保证只攻击靶机、不攻击生产机。

**可能答案 / 选项**：

| 选项 | 机制 | 强度 |
| --- | --- | --- |
| A | 黑名单（禁止攻击的 IP 列表） | 弱 — 一旦黑名单漏配则误攻击 |
| B | 白名单（只允许攻击列表内 IP） | 强 — 一旦白名单缺失则不能跑（推荐） |
| C | 双重确认：白名单 + 任务执行前再次校验 + 安委会审批 | 强（合规要求高时） |
| D | 物理隔离：靶机网段与生产网无 L2 / L3 互通 | 最强但工程量大 |

**项目方倾向**：选项 C（白名单 + 执行前强制校验）。生产 OT 网额外加选项 D（影子环境）。

**风险与权衡**：
- 🔥 **安委会审批是关键节点**：可能延期 2-4 周
- 误执行的应急流程：≤ 1 小时内回滚 + 安全事故报告

**评估证据**：
- 客户安委会的审批流程时长
- 历史是否有平台误执行事故案例
- 客户合规要求等级（等保 2.0 / 关基 KIIP）

**后续动作**：

## 3. 一般待澄清（试运行期补齐）

### 3.1 用户角色 / 权限模型

**待澄清**：（a）操作员 / 审计员 / 管理员 / 安委会人员的 RBAC 矩阵；（b）多用户并发协作（编辑同一攻击链路时的锁机制）；（c）操作日志审计粒度。

**Brainstorm**：默认 4 个角色（管理员 / 操作员 / 审计员 / 只读）+ RBAC AOP @RBAC（OpenAEV 已实现）。客户可能要求集成集团 IAM 角色映射 → 试运行期补适配。多用户协作的锁机制走 optimistic locking（已是 OpenAEV 默认）。

### 3.2 平台可用性 / 容灾要求

**待澄清**：（a）平台 SLA（99.9% / 99.99%）；（b）单机部署 vs HA 主备；（c）数据备份策略；（d）灾难恢复 RTO / RPO。

**Brainstorm**：默认 99.9% SLA + 单机部署。HA 主备需追加硬件（招标 §2 是 1 台 2U 机架），先按单机交付，HA 作为 follow-up。备份策略：PostgreSQL 每日全备 + WAL + Elasticsearch snapshot 至 S3。RTO 4 小时 / RPO 24 小时默认。

### 3.3 升级 / 维护窗口

**待澄清**：（a）平台升级窗口（如固定每月第 1 周末）；（b）回滚机制；（c）用例库更新通知机制（UI badge / 邮件）。

**Brainstorm**：默认每月第 1 周末 22:00-02:00 维护窗口。升级走 docker compose 滚动重启 + Flyway 自动迁移。回滚机制：DB 备份 + 容器镜像版本锁定。用例库更新通知走 UI badge + 邮件订阅（用例库变更后台 job）。

### 3.4 数据导入 / 导出格式标准

**待澄清**：（a）用例批量导入的 Excel / JSON 字段标准；（b）攻击编排链路的可导出格式；（c）跨平台兼容性（与集团其他验证平台）。

**Brainstorm**：默认 Excel 模板（用例批量导入）+ JSON 导出（链路 / 配置）。跨平台兼容性：OpenAEV 已支持 STIX 2.1 + ATT&CK 标准格式，与 MITRE 生态互通。导入临时表 `case_import_batch` → 派生 attack_chain（GAP §6.1 已列）。

### 3.5 报告输出的频率与渠道

**待澄清**：（a）实时大屏 / 周报 / 月报 的优先级；（b）邮件订阅机制；（c）API 对外开放范围。

**Brainstorm**：默认 dashboard 实时 + 月报 Word + 季报 Word + 突发事件即时报告。邮件订阅：每月 1 号自动发送月报。API 对外开放：限 RESTful read-only + OAuth 2.0 鉴权 + 速率限制。

### 3.6 性能基线

**待澄清**：（a）平台支持的最大并发任务数；（b）30 000 攻击组合在多长时间内跑完；（c）单 inject 的 P95 延迟要求；（d）Agent 心跳间隔与上报间隔。

**Brainstorm**：默认承诺指标 — （a）平台并发任务 ≥ 100；（b）30 000 组合在 4 小时内跑完（限流 50 并发）；（c）单 inject P95 延迟 ≤ 5 秒；（d）Agent 心跳 30 秒 / 上报 1 分钟。性能不达标需追加硬件（CPU + 内存）。

### 3.7 IPv6 单栈程度

**待澄清**：（a）平台与外部对接是否需 IPv4 兼容（NxSOC / CAPE / 邮件等）；（b）应用层是否纯 IPv6；（c）DNS / NTP / 镜像源是否双栈。

**Brainstorm**：默认平台内部纯 IPv6 + 外部对接（NxSOC / CAPE / SMTP / 集团 IPv6 平台）双栈。DNS / NTP / 镜像源用 IPv6（如阿里云镜像源有 IPv6 endpoint）。

### 3.8 培训教材的产权与延续

**待澄清**：（a）培训教材是否成为甲方资产；（b）培训后续 1 年内的复训机制；（c）线上 / 离线培训组合。

**Brainstorm**：教材项目方主编 + 甲方共享版权（甲方可二次分发）。培训后 1 年内 ≤ 2 次复训免费。线上培训用 Zoom / 腾讯会议 + 录像存档。

### 3.9 攻击特征库更新机制（招标交付文档 #4）

**待澄清**：（a）特征库更新频率（每周 / 每月）；（b）更新来源（社区 CVE / 商业威胁情报）；（c）甲方审核流程；（d）离线更新方案。

**Brainstorm**：默认每月 1 次特征库更新（用例库 / Nuclei / ART 同步）+ 紧急 CVE 24 小时内补充。更新来源：MITRE ATT&CK + Atomic Red Team + Nuclei templates + 国产 PoC 库。离线更新：项目方提供升级包，甲方人工导入。

### 3.10 多区域部署中的网络拓扑细节
